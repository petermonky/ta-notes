Encapsulation

The bundling of data and operations into a single unit.

A way to isolate lower-level implementations from the client — creates an abstraction barrier. 


          class Circle {
            double x;
            double y;
            double r;

            double getArea() {
              return 3.141592653589793 * r * r;
            }
          }

Information Hiding

The mechanism for restricting access to some or all of a unit's components.

Prevents the client from making assumptions about the internal representation of a class. 


          class Circle {
            private double x;
            private double y;
            private double r;

            public double getArea() {
              return 3.141592653589793 * r * r;
            }
          }

Tell Don't Ask

The enforcement of encapsulation and information hiding.

Perform operations without the knowledge of the internal representation a class. 


          // don't ask...
          double cX = c.getX();
          double cY = c.getY();
          double r = c.getR();
          boolean isInCircle =
            ((x - cX) * (x - cX) + (y - cY) * (y - cY)) <= r * r;

          // ...tell!
          boolean isInCircle = c.contains(x, y);

Inheritance

The mechanism by which one class inherits the data and operations of another class.

Allows for the implementation of an IS-A relationship.


          class ColouredPoint extends Point {
            private Colour colour;
          
            public ColouredPoint(double x, double y, Colour colour) {
              super(x, y);
              this.colour = colour;
            }
          }

Polymorphism

The apparent transformation of existing code in a run-time context.

Inspired by method overriding — method signature must be taken into account. 


          class Circle {
            :
            @Override
            public boolean equals(Object obj) {
              if (obj instanceof Circle) {
                Circle circle = (Circle) obj;
                return (circle.c.equals(this.c) && circle.r == this.r);
              }
              return false;
            }
          }

Lab 1

What if we want to hide the internal representation?

ShopSimulation has publicly declared fields, and so can be changed by the client.

How can this implementation be hidden away?

(Hint: Information Hiding — Access modifiers!)

Lab 1

What if we want to add a new event type?

ShopEvent is very inefficient; manually checks the event type.

How can we make the simulation more scalable as new event types are introduced?

(Hint: Inheritance, Polymorphism — Replace ShopEvent with classes for each individual event.)

Lab 1

What if we want to change the simulation logic?

It is challenging to extend the current code with new simulation logic (e.g., new event types or new customer behaviour).

How can each component of the simulation be compartmentalised?

(Hint: Encapsulation, Tell Don't Ask — A shop simulation involves a shop, customers, and counters.)
